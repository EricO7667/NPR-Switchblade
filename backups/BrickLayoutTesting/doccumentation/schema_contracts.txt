====================================================================
NPR SWITCHBLADE — SCHEMA CONTRACTS & MENTAL MODEL
====================================================================
Purpose:
--------
This document defines how SCHEMAS are used in the NPR Switchblade
pipeline, what they mean, and how developers are expected to reason
about them when writing or modifying bricks.

Schemas are a HUMAN-FIRST contract system.
They are NOT runtime validators by default.

====================================================================
1. WHAT A SCHEMA IS
====================================================================

A schema is a semantic label attached to a Payload that describes
what the data is supposed to look like at a specific point in the
pipeline.

A schema answers one question:

    "If I inspect payload.data right now, what should I expect?"

Schemas:
- Describe pipeline stage and data shape
- Communicate assumptions between bricks
- Serve as contracts between steps

Schemas do NOT:
- Enforce validation automatically
- Guarantee correctness at runtime
- Replace unit tests or debugging

====================================================================
2. HOW SCHEMA CONTRACTS WORK
====================================================================

Each schema has a documented contract consisting of:

- description
- guarantees
- may_contain
- must_not_contain
- allowed_next

These contracts define what downstream bricks are allowed to assume.

Example:

    guarantees:
      - resistance_ohms exists and is numeric

This means:
- Every row MUST have resistance_ohms at this stage
- Downstream bricks may safely access it
- If it is missing, the pipeline is logically broken upstream

====================================================================
3. GUARANTEES VS MAY_CONTAIN
====================================================================

GUARANTEES:
-----------
Fields or properties that MUST exist for ALL rows.

If a downstream brick cannot function without a field, that field
MUST be listed in guarantees.

Example:
- resistance_ohms
- component_type

Downstream bricks may assume these exist WITHOUT defensive checks.

MAY_CONTAIN:
------------
Optional fields that may exist for some rows, but are not required.

Example:
- tolerance_percent
- voltage_rating
- wattage

Downstream bricks must NOT rely on these without checking.

====================================================================
4. HOW BRICKS USE SCHEMAS
====================================================================

Every brick follows the same mental model:

1. Receive a Payload with schema A
2. TRUST the schema A contract
3. Transform the data
4. Output a new Payload with schema B

Bricks should NOT:
- Guess what fields exist
- Defensively check guaranteed fields
- Patch missing data silently

If a guaranteed field is missing:
- That is a DESIGN BUG
- The fix is upstream, not defensive code

====================================================================
5. SCHEMAS ARE NOT VALIDATION
====================================================================

Schemas do not validate data by default.

They exist to:
- Make assumptions explicit
- Prevent silent bugs
- Enable clean brick logic
- Allow optional enforcement later

Optional validation (if ever added) should live in:
- Dedicated validator bricks
- Debug-only checks
- CI or development tooling

Never in the core Payload or Runner.

====================================================================
6. SCHEMA EVOLUTION RULES
====================================================================

- Schema names represent PIPELINE STAGES, not logic
- Do NOT encode logic in schema names
  (bad: matched_results_exact)
- Use metadata or fields for classification instead

Schemas should be:
- Few
- Stable
- Boring

====================================================================
7. ONE-SENTENCE RULE (AUTHORITATIVE)
====================================================================

Schemas describe what data looks like at a point in time.
Bricks trust the schema they receive and must produce data that
matches the schema they emit.
